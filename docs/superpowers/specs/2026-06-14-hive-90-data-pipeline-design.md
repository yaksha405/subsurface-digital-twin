# HIVE 90+ Data Pipeline And Product Hardening Design

> **文档级别**：产品/架构规格（Specification）
>
> **适用范围**：HIVE 90+ 产品升级，覆盖 mock 数据底座、场景语义、接口容错、交互一致性、全量验收
>
> **对应代码真源**：`src/api/` `src/data/` `src/hooks/` `src/lib/sceneSemantics.ts` `src/components/` `backend/`

---

## 1. 背景

当前 HIVE 已具备较完整的工业演示骨架：多场景切换、3D 场景、角色面板、AI 助手、导出、证据链、基础国际化均已存在。但现阶段仍存在三个根本问题：

1. 面板结果仍有“展示先行、数据后补”的倾向，容易造成跨模块数字不一致。
2. 不同场景仍可能出现术语、参数、阈值、图层名称串场。
3. 真实研发数据接入时，格式、命名、单位、质量、缺失值都可能与当前假设不同，如果接口层写死，将导致后续返工。

因此，本次升级不以“继续补 UI 数字”为核心，而以“建立接近真实系统的数据链路和产品约束”为核心。

---

## 2. 产品目标

将 HIVE 提升为一个可用于客户演示、学术交流、方案讲解的 90 分以上工业级演示产品。

该目标的定义不是“功能更多”，而是：

- 数据逻辑自洽
- 场景语义一致
- 关键交互稳定
- AI 与导出具有工程可信边界
- 对未来真实数据接入具备容错性和可扩展性

---

## 3. 设计原则

### 3.1 原始数据优先

mock 的对象必须是原始数据和中间分析结果，而不是最终面板数字。

正确链路应为：

`原始观测/设备状态/事件流/分析结果 -> API 适配与标准化 -> 场景语义映射 -> 派生指标计算 -> UI/导出/AI 展示`

### 3.2 单一事实来源

同一个统计口径在系统中只能有一个权威来源。任何卡片、列表、详情、导出、AI 摘要都应从同一计算链派生，而不是分别手填。

### 3.3 场景隔离

煤矿、金矿、油气、管线、核设施、炼化、地下暗流必须被视为不同的业务上下文，而不是“同一 UI 壳上换标题”。

### 3.4 接口容错与可扩展

接口层必须假设未来研发提供的数据会出现以下变化：

- 字段名变化
- 单位变化
- 枚举值变化
- 缺失字段
- 冗余字段
- 数据质量波动
- 时间戳/坐标格式差异

因此，前端与 mock/真实 API 之间必须增加可配置、可退化、可告警的适配层，而不是写死某一种标准。

### 3.5 工业可信边界

AI、预测、插值、推断必须明确显示其证据边界，不允许伪装成“已确认事实”。

---

## 4. 范围

### 4.1 本次纳入

- 统一 mock 原始数据模型
- API 标准化与容错适配
- 场景语义配置统一
- 关键指标派生计算统一
- 数据一致性自动校验
- 关键交互稳定性修复
- 中英文全量接管
- 文档索引与验收清单体系

### 4.2 本次不纳入

- 真正的设备侧采集 SDK
- 企业级权限体系与账号系统
- 多人实时协作后端
- 重型 CAD/GIS/点云专业分析替代

HIVE 依然定位为“轻量、可信、可交付的未知空间安全认知工作台”，而不是重型工业平台。

---

## 5. 目标架构

目标架构分为六层：

1. **Raw Mock Layer**
   - 模拟真实设备与传感器回传
   - 模拟事件流、告警流、任务流、分析结果

2. **Contract Adapter Layer**
   - 负责容错解析、字段映射、默认值处理、单位标准化、枚举兼容
   - 同时适配 mock 和未来真实 API

3. **Semantic Layer**
   - 为每个场景定义指标、单位、阈值、标签、对象类型、AI 话术、导出语言

4. **Derived Metrics Layer**
   - 统一生成覆盖率、告警统计、机器人统计、趋势聚合、风险等级、导出摘要

5. **Presentation Layer**
   - UI、右侧详情、导出、AI 可视结果全部消费派生结果

6. **Verification Layer**
   - 自动测试 + 手工验收清单，保证数值、文案、交互、切换的一致性

## 6. 数据底座设计

### 6.1 Raw Mock 数据类型

统一 mock 原始数据至少包含：

- `robotTelemetry`
  - 机器人身份、型号、角色、位置、电量、链路质量、任务状态、健康状态
- `sensorSamples`
  - 时间序列采样点，包含坐标、采样来源、物理量、单位、置信度
- `spatialAssets`
  - 裂缝、管段、反应堆构件、炼化设备通道、地下暗流通道等对象
- `events`
  - 告警、断联、低电量、异常温度、异常压力、维护事件、人工确认事件
- `missionRecords`
  - 探测任务、覆盖里程、阶段状态、交付状态、时间线片段
- `analysisOutputs`
  - AI 推断、插值结果、风险分级、候选关注点、证据边界

### 6.2 禁止事项

以下内容不得作为独立手填真源长期存在：

- 左侧概览卡统计数字
- 传感器趋势卡的最终聚合结果
- 告警计数
- 机器人总数/在线数/低电量数
- 导出报告中的摘要数值

这些都必须由原始数据或分析输出统一派生。

## 7. 接口容错与可扩展设计

### 7.1 容错目标

系统应能接受“非完全理想”的研发数据输入，而不至于整页崩溃或逻辑失效。

### 7.2 适配层能力要求

接口适配层必须具备：

1. **字段别名映射**
   - 例如 `battery` / `batteryLevel` / `power_pct` 可映射到统一内部字段

2. **枚举兼容**
   - 例如 `critical`、`danger`、`high_risk` 等可映射到统一风险级别

3. **单位标准化**
   - 例如 `kPa` 与 `MPa`、`ppm` 与 `%`、`C` 与 `°C`

4. **默认值与降级策略**
   - 缺字段时使用 `unknown`、`null`、受控 fallback，而不是伪造确定值

5. **宽松读取、严格输出**
   - 输入尽量兼容，内部结构标准化，输出给组件的结构保持稳定

6. **质量标记**
   - 对缺失、异常、推断、插值数据附带 `quality` / `truthBoundary` 标识

7. **非阻塞失败**
   - 某个局部模块数据异常时，不应拖垮整个页面；应局部降级并留下可见提示或日志

### 7.3 扩展目标

未来新增场景、新字段、新分析模型时，预期只需：

- 新增场景语义配置
- 新增字段映射规则
- 新增派生计算器

而不是修改大量组件条件分支。

---

## 8. 场景语义设计

每个场景必须集中定义以下信息：

- 对象名称
- 指标名称
- 指标单位
- 风险阈值
- 图层名称
- 右侧详情字段
- 快捷分析命令
- AI 响应话术
- 导出说明文本
- 面板标题/副标题

目标是做到：

1. 地下暗流不再出现裂缝区、瓦斯词汇。
2. 核设施场景不再沿用岩层/裂缝网络命名。
3. 油气、金矿、煤矿共享同类结构，但各自拥有独立参数与风险语言。

---

## 9. 派生计算设计

系统内所有关键业务数字都应通过统一派生层生成，包括但不限于：

- 覆盖率
- 已采样节点 / 未探明节点
- 活跃告警数 / 紧急告警数 / 未处理数
- 在线/离线/低电量/故障机器人
- 趋势聚合
- 风险优先级
- 导出就绪度
- 任务进展状态

派生规则必须可测试、可追溯、可解释。

每个关键数值都应能回答：

1. 来源于哪些原始数据？
2. 采用了什么规则？
3. 当前是否包含插值/AI 推断？

---

## 10. 交互硬化设计

本次不只修视觉，更要修“操作可信感”。

重点包括：

- 3D 旋转、平移、缩放稳定
- 点对象后右侧详情稳定联动
- 工具进入/执行/退出路径清晰
- 测距、框选、剖面、标注有命中反馈
- 切换场景/切换语言/切换工具不残留脏状态

对产品经理与真实用户的解释标准是：

- 为什么现在要让用户点这里？
- 点完之后系统发生了什么？
- 如果当前场景不支持这个操作，是否明确告诉了用户？

---

## 11. 文档与索引要求

本项目后续所有规划必须落档，并进入索引。

要求如下：

1. 规格文档统一放在 `docs/superpowers/specs/`
2. 计划文档统一放在 `docs/superpowers/plans/`
3. 产品标准索引 `docs/README.md` 必须登记
4. 根 README 中的文档总览也必须登记
5. `task_plan.md`、`findings.md`、`progress.md` 持续更新

目标是避免未来重复调研、重复规划、重复踩坑。

## 12. 阶段计划

### 阶段 1：统一数据底座

目标：建立统一原始 mock 数据管线，去掉手填式结果依赖。

验收：

- 面板关键数字全部可追溯
- 组件不直接依赖零散 mock 常量
- 三个以上主场景跑通统一取数链路

### 阶段 2：接口适配与容错

目标：让系统能容纳真实研发数据标准差异。

验收：

- 存在字段别名映射机制
- 存在枚举兼容和单位标准化
- 缺失字段不会导致页面整体崩溃
- 不同质量数据有明确标识

### 阶段 3：场景语义统一

目标：每个场景整页语义自洽。

验收：

- 场景切换后整页术语、阈值、图层、详情同步更新
- 无明显跨场景串词

### 阶段 4：一致性自动校验

目标：确保系统内数字不打架。

验收：

- 统计卡与列表一致
- 覆盖率与采样点一致
- 告警数与告警列表一致
- 机器人统计与列表一致

### 阶段 5：交互修复与增强

目标：达到可演示级稳定交互。

验收：

- 点对象、切场景、切工具、做测量稳定可用
- 不再出现明显空白、错联动、残留状态

### 阶段 6：国际化与工业级 polish

目标：整体验收达到国际工业演示水准。

验收：

- 中英文全量切换
- 关键页面无占位文案与错词
- AI/导出/证据链同源且可信

### 阶段 7：全量验收

目标：达到 90+ 演示标准。

验收：

- 无 P0/P1 阻塞问题
- 关键链路全通
- 数据、术语、交互经得起现场追问

---

## 13. 90+ 验收总标准

### A. 数据真实性

- 100% 关键数值有来源和计算逻辑
- 不允许明显逻辑矛盾

### B. 场景一致性

- 任何场景都不出现明显串词、错单位、错阈值

### C. 交互稳定性

- 核心演示链路稳定，不因常规点击而出 bug

### D. 工业可信感

- AI 有边界
- 导出有依据
- 面板有角色感与证据感

### E. 可接真实数据

- 接口层可适配不同标准，而不是写死一套样板格式

---

## 14. 风险

主要风险包括：

1. 现有组件内部仍混有历史直出逻辑，拆分时可能暴露隐藏耦合。
2. mock 数据重构可能影响既有视觉快照和部分测试。
3. 如果不先建立字段来源表，后面仍可能局部“补数字”回潮。

应对策略：

- 先做数据来源盘点
- 再改适配层和派生层
- 最后统一收 UI

---

## 15. 结论

本次升级的核心不是“继续修界面细节”，而是把 HIVE 的数据链、语义链、交互链和验收链全部做实。

当该设计完成后，HIVE 将具备以下能力：

- 像真实工业系统一样从源数据驱动展示
- 对未来研发数据接入更友好
- 更不容易在演示、评审、研讨会中穿帮
- 更接近一个可商业化的国际工业产品，而不是一个会动的原型壳
